Statistics of trend store parts
-------------------------------

table:
trend_directory.trend_store_part_stats

columns:
* trend_store_part_id > foreign key to trend_store_part.id
* timestamp
* modified
* count

primary key is the pair (trend_store_part_id, timestamp).
count is the number of data values that is present for that trend_store_part and that timestamp
modified is the point in time at which the count was calculated

necessary functions:

* flagging function
trend_directory.find_trend_store_part_stats_to_update()
provides a list of trend_store_part_stats
attempted definition (either untested code or pseudocode):
SELECT tsps.* (or maybe tsps.trend_store_part_id, tsps.timestamp)
  FROM trend_directory.trend_store_part_stats tsps
  JOIN trend_directory.modified m
  ON tsps.trend_store_part_id = m.trend_store_part_id
    AND tsps.timestamp = m.timestamp
  WHERE tsps.modified < m.last

* update function
trend_directory.recalculate_trend_store_part_stats(trend_store_part_id, timestamp)
updates a trend_store_part stat
attempted definition:
EXECUTE format('UPDATE trend_directory.trend_store_part_stats
  SET updated, count = now(), COUNT(*)
    FROM trend.%I WHERE timestamp = %2', SELECT ts.name FROM trend_store_part ts WHERE ts.id = %1)

* complete check
trend_directory.recalculate_changed_trend_store_part_stats()
attempted definition:
SELECT trend_directory.recalculate_trend_store_part_stats(tsps.trend_store_part_id, tsps_timestamp)
  FROM trend_directory.find_trend_store_part_stats_to_update() tsps

* create function
I propose to do creation as a trigger on creation on the modified table.
trigger condition:
AFTER INSERT ON trend_directory.modified FOR EACH ROW
attempted definition:
INSERT INTO trend_directory.trend_store_part_stats (trend_store_part_id, timestamp, last, count)
  VALUES (NEW.trend_store_part_id, NEW.timestamp, now(), ***);
where the *** defined as in the update function(presumably included through a 'WITH' mechanism)

* transitional measure
To get from a system *without* these stats to a system *with* these stats, the stats will preferably be calculated for old data too, so the system can get into full swing as soon as possible. The function will look much like the trigger being run for the whole modified table, with an "ON CONFLICT DO NOTHING" added to it.
